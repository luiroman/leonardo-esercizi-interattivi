INIZIO Situazione iniziale

Client e server non si conoscono. Il canale è in chiaro — chiunque può leggere il traffico. Obiettivo: stabilire un canale cifrato in un solo round-trip.

1 Client Hello → Server

Il client genera una coppia di chiavi Diffie-Hellman effimere (usa e getta) e invia la chiave pubblica al server insieme a: versioni TLS supportate, cipher suites proposte, e il nome del sito (SNI). Inviare la chiave DH già nel primo messaggio è la novità di TLS 1.3 — permette di completare in 1-RTT.

2 Server Hello + Certificate ← Server

Il server sceglie la cipher suite, genera la propria chiave DH effimera, e la invia insieme al certificato X.509 (che prova la sua identità). Il messaggio Finished è già cifrato — il server ha calcolato il session key usando la chiave DH del client + la propria.

3 Client Finished → Server (cifrato!)

Il client verifica il certificato risalendo la catena di fiducia fino alla Root CA. Poi calcola lo stesso session key (DH pubblica server + DH privata client = stessa chiave!) e invia il Finished cifrato. Da questo momento, tutto è cifrato.

🔒 Connessione sicura stabilita!

Client e server condividono lo stesso session key senza averlo mai trasmesso. Anche se qualcuno ha intercettato tutto, non può ricostruire la chiave. Le chiavi DH sono effimere — verranno distrutte a fine sessione (Perfect Forward Secrecy).
