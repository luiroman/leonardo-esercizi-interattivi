2. Routing: Come i Pacchetti Trovano la Strada

La tabella di routing

Ogni router mantiene una tabella di routing — una struttura dati che associa prefissi di rete (destinazioni) a next-hop (il prossimo passo per raggiungere quella rete). Quando un pacchetto arriva, il router consulta la tabella e cerca la rotta più specifica che corrisponda alla destinazione.

Longest Prefix Match
Tra tutte le rotte che corrispondono alla destinazione, il router sceglie sempre quella con il prefisso più lungo (più specifico). Un pacchetto diretto a 10.0.1.50 matcherà /24 piuttosto che /8 o /0. La default route (0.0.0.0/0) ha il prefisso più corto possibile e viene usata come ultima risorsa.

Le rotte nella tabella possono avere diverse origini, identificate da una lettera: C (Connected — reti direttamente collegate alle interfacce del router), S (Static — configurate manualmente), O (OSPF — apprese via protocollo dinamico), B (BGP — apprese via protocollo esterno). Le rotte Connected appaiono automaticamente quando si configura un indirizzo IP su un'interfaccia.

Routing statico

Il routing statico consiste nel configurare manualmente ogni rotta nella tabella del router. L'amministratore decide esattamente quale percorso seguirà ogni pacchetto.

VantaggiSvantaggi
Controllo totale sul percorsoNon scala con reti grandi
Nessun overhead di protocolloNessun failover automatico
Più sicuro (nessun annuncio di rotte)Errori umani frequenti
Semplice da capire e debuggareManutenzione manuale ad ogni cambio

Si usa principalmente in reti piccole (meno di 10 sottoreti), per la default route verso l'ISP, per rotte specifiche di sicurezza (es: traffico sensibile via tunnel VPN), e come floating static route — una rotta statica di backup con metrica più alta della rotta dinamica.

# Aggiungere una rotta statica su Linux
ip route add 10.0.2.0/24 via 10.0.1.1

# Default route
ip route add default via 203.0.113.1

Routing dinamico

I protocolli di routing dinamico permettono ai router di scambiarsi automaticamente informazioni sulle reti raggiungibili. Quando un link cade, i protocolli ricalcolano le rotte senza intervento umano. Si dividono in tre famiglie: Distance Vector (RIP — obsoleto), Link State (OSPF, IS-IS), e Path Vector (BGP).

3. OSPF — Open Shortest Path First

OSPF è il protocollo IGP (Interior Gateway Protocol) link-state più utilizzato nelle reti enterprise. Ogni router costruisce una mappa completa della topologia della rete e calcola indipendentemente i percorsi ottimali.

Come funziona in 5 passi

  1. Discovery: i router scoprono i vicini scambiando pacchetti Hello a intervalli regolari (default 10s su Ethernet).
  2. Exchange: i router si scambiano Database Description (DBD) per sincronizzare i loro LSDB.
  3. Database: ogni router costruisce il Link State Database (LSDB), una mappa completa della topologia.
  4. Calcolo: l'algoritmo SPF (Dijkstra) calcola l'albero dei percorsi minimi con il router come radice.
  5. Routing: i percorsi ottimali vengono inseriti nella tabella di routing.
Concetti chiave OSPF
Area 0 è il backbone — obbligatorio, tutte le aree devono connettersi ad esso. Il costo OSPF è inversamente proporzionale alla bandwidth (reference bandwidth / interface bandwidth). Il DR/BDR (Designated Router / Backup DR) ottimizzano le adiacenze su reti broadcast. La convergenza è rapida: 1-5 secondi.
🔒 Sicurezza OSPF
OSPF è vulnerabile a router rogue che annunciano rotte false (rogue router, LSA injection). La difesa è l'autenticazione OSPF con MD5 o SHA-256 (RFC 5709) — ogni pacchetto OSPF deve essere firmato con una chiave condivisa.

4. BGP — Border Gateway Protocol

BGP è l'unico protocollo EGP (Exterior Gateway Protocol) in uso su Internet. Mentre OSPF gestisce il routing dentro un'organizzazione, BGP gestisce il routing tra le grandi reti (Autonomous Systems) che compongono Internet. Ogni ISP, cloud provider e grande azienda è un AS con un proprio numero (ASN).

Le decisioni di routing BGP non si basano sulla distanza, ma su un insieme di attributi e policy configurate dall'operatore. La tabella BGP globale contiene circa 950.000 prefissi IPv4 — il che dà un'idea della scala a cui BGP opera.

TipoDove operaCaratteristiche
eBGP (external)Tra AS diversiTCP porta 179, TTL=1 (vicini diretti)
iBGP (internal)Dentro lo stesso ASFull mesh o Route Reflector per scalare
⚠ BGP Hijacking
Un attaccante può annunciare prefissi BGP che non gli appartengono, dirottando il traffico verso la propria rete. Casi celebri: Pakistan Telecom vs YouTube (2008), China Telecom (2018). La difesa è RPKI (Resource Public Key Infrastructure) che valida crittograficamente l'origine dei prefissi.

In pratica: strategia enterprise

In un'azienda come Leonardo, la strategia tipica è: OSPF per il routing interno (campus, data center, sedi) + BGP per la connettività esterna (verso ISP e cloud provider). Le rotte statiche restano per casi specifici: default route, tunnel VPN, rotte di management.

5. Workshop Wireshark

Wireshark è l'analizzatore di protocolli più utilizzato al mondo. Permette di catturare e analizzare il traffico in tempo reale o da file .pcap. L'interfaccia ha tre aree: Packet List (elenco dei pacchetti), Packet Details (scomposizione in livelli), e Packet Bytes (contenuto grezzo).

Esercizi svolti in aula

TCP Three-Way Handshake

Filtro: tcp.flags.syn == 1. Il handshake si compone di tre pacchetti: SYN (client → server, seq=X), SYN-ACK (server → client, seq=Y, ack=X+1), ACK (client → server, ack=Y+1). Il delta time tra SYN e ACK finale misura il RTT (Round Trip Time).

DNS Query e Response

Filtro: dns. Le query DNS viaggiano su UDP porta 53 in chiaro — il campo Queries contiene il dominio richiesto, la response contiene l'indirizzo IP con il TTL.

HTTP vs HTTPS

Con http vedi tutto in chiaro: URL, header, cookie, body. Con tls vedi solo Client Hello (con SNI), Server Hello, e dati cifrati. Questa è la dimostrazione più efficace del perché HTTPS è essenziale.

🔒 ARP Spoofing
L'ARP spoofing permette a un attaccante di associare il proprio MAC all'IP del gateway, intercettando tutto il traffico (Man-in-the-Middle). Segnali in Wireshark: stesso IP con MAC diversi, ARP Reply non richiesti. Difesa: Dynamic ARP Inspection (DAI) sugli switch managed.

Funzioni statistiche

Quattro funzioni utili per l'analisi rapida: Protocol Hierarchy (distribuzione dei protocolli), Conversations (coppie di host comunicanti), Endpoints (inventario degli host), I/O Graph (traffico nel tempo). Tutte accessibili dal menu Statistics.

6. DNS in Profondità

Gerarchia e risoluzione

Il DNS è un sistema distribuito e gerarchico. Quando digiti un dominio, il resolver esegue fino a 4 query iterative risalendo la gerarchia: Root Servers (13 gruppi A-M, distribuiti globalmente via anycast) → TLD Servers (.com, .it, .org) → Authoritative Name Server (gestisce il dominio specifico) → risposta con l'IP e il TTL per il caching.

Tipi di record

TipoFunzioneUso Security
ANome → IPv4Risoluzione standard
AAAANome → IPv6Dual-stack
CNAMEAlias (nome canonico)CDN, load balancing
MXMail server per il dominioEmail spoofing, SPF
TXTTesto arbitrarioSPF, DKIM, DMARC
NSName server autorevoleDomain hijacking
PTRIP → nome (reverse DNS)Verifica identità, anti-spam
SRVServizio + portaService discovery (Active Directory)

Attacchi DNS

Il DNS è uno dei servizi più attaccati di Internet. I principali attacchi sono: DNS Spoofing (risposte false iniettate nella cache), DNS Tunneling (dati nascosti nelle query per exfiltration o C2), DNS Amplification (DDoS volumetrico), Domain Hijacking (modifica dei record NS) e Typosquatting (domini simili per phishing).

Difese DNS

DNSSEC (RFC 4033-4035) aggiunge firme crittografiche ai record DNS — il resolver può verificare autenticità e integrità, ma DNSSEC non cifra le query. Per la privacy servono DNS over HTTPS (DoH, RFC 8484 — porta 443, indistinguibile dal traffico web) o DNS over TLS (DoT, RFC 7858 — porta 853).

💡 Impatto enterprise di DoH
DoH è problematico per il SOC perché rende il traffico DNS invisibile (cifrato dentro HTTPS sulla porta 443). La best practice enterprise è: DNS interno con logging completo → log al SIEM, blocco DoH/DoT verso resolver esterni.

7. SSL/TLS e PKI

TLS 1.3 — L'handshake moderno

TLS (Transport Layer Security) è il protocollo crittografico che protegge le comunicazioni su rete. TLS 1.3 (RFC 8446) è la versione corrente, significativamente migliorata rispetto a TLS 1.2.

L'handshake TLS 1.3 si completa in 1 round-trip (1-RTT), contro i 2-RTT di TLS 1.2:

  1. Client Hello: il client invia versioni supportate, cipher suites, e la chiave pubblica effimera Diffie-Hellman.
  2. Server Hello + Certificate: il server sceglie la cipher suite, invia la sua chiave DH effimera, il certificato X.509 e il messaggio Finished cifrato.
  3. Client Finished: il client verifica il certificato, calcola il session key condiviso (ECDHE), invia Finished cifrato.
Perfect Forward Secrecy (PFS)
TLS 1.3 impone la PFS usando solo scambi di chiave effimeri (ECDHE). Ogni sessione ha chiavi temporanee che vengono distrutte alla fine — se qualcuno compromette la chiave privata del server, non può decifrare le sessioni passate. TLS 1.3 ha anche rimosso tutti gli algoritmi deboli: niente RSA key exchange, CBC, RC4, SHA-1.

Certificati X.509 e catena di fiducia

Un certificato X.509 è la "carta d'identità digitale" di un server. Contiene il Subject (identità), l'Issuer (CA che l'ha firmato), la validità temporale, la chiave pubblica, i Subject Alternative Names (SAN — i domini per cui è valido), e la firma della CA.

La catena di fiducia funziona così: il certificato del server è firmato da un'Intermediate CA, che a sua volta è firmata da una Root CA. La Root CA è autofirmata e presente nel trust store del sistema operativo (~150 Root CA). Il browser verifica ogni firma risalendo la catena.

⚠ Errori comuni con i certificati
Certificato scaduto — il campo Not After è superato. Let's Encrypt offre certificati gratuiti a 90 giorni con rinnovo automatico (ACME).
Nome non corrispondente — il CN/SAN non include il dominio richiesto.
Catena incompleta — il server non invia l'Intermediate CA.

8. VPN — Virtual Private Network

Una VPN crea un tunnel cifrato su una rete pubblica (Internet), estendendo una rete privata in sicurezza. Fornisce tre servizi: confidenzialità (cifratura), integrità (protezione contro modifiche) e autenticazione (verifica dell'identità).

Site-to-site vs Remote Access

Site-to-SiteRemote Access
Cosa collegaDue reti aziendaliUn singolo utente alla rete
EsempioRoma ↔ Napoli, Sede ↔ AzureDipendente casa → rete Leonardo
TunnelAlways-on, gateway-to-gatewayOn-demand, client-to-gateway
ProtocolloIPSec (IKEv2)SSL VPN o IKEv2

IPSec

IPSec opera al Layer 3 ed è composto da: ESP (Encapsulating Security Payload — cifratura + autenticazione), AH (Authentication Header — solo autenticazione, poco usato), e IKE/IKEv2 (negoziazione delle chiavi, UDP 500/4500). IPSec supporta due modalità: tunnel mode (cifra l'intero pacchetto, usato per site-to-site) e transport mode (cifra solo il payload, usato host-to-host).

SSL VPN vs IPSec VPN

La SSL VPN opera ai Layer 4-7 usando TLS su porta TCP 443 — funziona ovunque ci sia accesso HTTPS (anche in hotel o aeroporti con WiFi restrittivi). IPSec è più performante (UDP) ma richiede porte dedicate che possono essere bloccate. In pratica: IPSec per site-to-site, SSL VPN per remote access.

🔒 Best practice VPN enterprise
MFA obbligatorio per remote access. Full tunnel per dipendenti (visibilità SOC). Certificati X.509 al posto delle pre-shared keys. Logging VPN integrato nel SIEM. Patching tempestivo dei VPN gateway (bersagli critici: vulnerabilità Fortinet, Ivanti, Citrix). Network segmentation post-VPN (least privilege).
💡 ZTNA — Il futuro oltre la VPN
Zero Trust Network Access (ZTNA) sta sostituendo la VPN tradizionale: invece di dare accesso all'intera rete, ZTNA verifica identità e stato del dispositivo prima di concedere accesso a singole applicazioni. Prodotti: Zscaler, Cloudflare Access, Azure AD Application Proxy.

11. Strumenti da Installare

pfSense su VirtualBox

  1. Scarica l'ISO di pfSense CE da pfsense.org/download (AMD64, ISO)
  2. In VirtualBox, crea una nuova VM: tipo FreeBSD (64-bit), 1GB RAM, 8GB disco
  3. Aggiungi 2 interfacce di rete: Adapter 1 = NAT (WAN), Adapter 2 = Host-only (LAN)
  4. Avvia la VM dall'ISO e segui l'installazione (accetta le impostazioni predefinite)
  5. Al riavvio, pfSense assegna automaticamente le interfacce: WAN (DHCP) e LAN (192.168.1.1)
  6. Dal browser del tuo host, vai a http://192.168.1.1 — login: admin / pfsense
💡 Troubleshooting pfSense
Se il browser non raggiunge 192.168.1.1: verifica che l'host-only network di VirtualBox sia attiva (File → Host Network Manager), e che il tuo host abbia un IP nella rete 192.168.1.x sull'interfaccia host-only.

Ispezione certificati con OpenSSL

openssl s_client -connect www.leonardo.com:443 \
  -servername www.leonardo.com < /dev/null 2>/dev/null | \
  openssl x509 -text -noout
